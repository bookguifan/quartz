# Markdown 输出格式规范

> **适用范围**：AI 增强后的学习笔记 Markdown 文件输出。
> **前置依赖**：必须先完成概念完整性核对、审核纠错、补齐增强。

---

## 一、输出格式骨架

### 1.1 文件命名

```
N.主题名（AI增强版）.md
```

- `N`：数字序号（对应原笔记顺序，如 1, 2, 3...）
- `主题名`：核心主题，如"机器学习概述"、"KNN算法"
- 示例：`1.机器学习概述（AI增强版）.md`

### 1.2 头部信息（YAML Frontmatter + 标题）

增强版笔记必须包含 YAML frontmatter，作为 Obsidian Properties 的数据源。frontmatter 位于文件最开头，用 `---` 分隔。

```markdown
---
title: 主题名（AI增强版）
date: [YYYY-MM-DD 或留空]
study_time: [时长或留空]
review_version: [当前Skill版本号]
publish: false
category: [大分类]
difficulty: [🔹基础 / 🔸核心 / 🔺难点]
tags:
  - 学习笔记
  - [主题标签]
  - [子标签...]
status: 完善
---
# [dayXX] 学习笔记｜主题名（AI 增强版）
```

**属性说明**：

| 属性 | 类型 | 取值规则 |
|------|------|---------|
| `title` | Text | 从 `#` 标题提取，去掉序号前缀（如 `[day02]`） |
| `date` | Date | 有则填 YYYY-MM-DD，无则留空（不填"未标注"） |
| `study_time` | Text | 有则填，无则留空 |
| `review_version` | Text | 与当前 Skill 版本完全一致 |
| `publish` | Checkbox | 默认 `false` |
| `category` | Text | 从文件夹名或标签体系推断（如"深度学习""机器学习"） |
| `difficulty` | Select | 根据笔记内难度标签分布判定：含🔺→`🔺难点`，否则含🔸→`🔸核心`，否则→`🔹基础` |
| `tags` | List | 从原笔记标签行提取，转为 YAML 列表格式（每项前加 `- `） |
| `status` | Select | 新生成默认 `完善`；若原笔记过于简略则用 `草稿` |

**注意事项**：
- 无法推断的日期/时长留空（不填"未标注"），在"8️⃣ AI 附加说明"中提示用户补充
- frontmatter 和 `#` 标题之间**禁止保留**旧的行内元数据行（`**📅 日期**`、`**⏱ 学习时长**`、`**🔧 AI 审核版本**`、`**🏷 标签**`）
- tags 必须使用 YAML 列表格式（`- 标签名`），不使用行内数组 `[a, b]`
- 属性值含特殊字符（冒号、# 等）时用双引号包裹

### 1.3 模块顺序（固定）

```markdown
## 📌 核心速览

> [!summary] 核心速览
> - **概念A**：一句话定义/描述（≤80字）。
> - **概念B**：一句话定义/描述（≤80字）。
> - **概念C**：一句话定义/描述（≤80字）。

**格式规则**：
- 概念数量：**5-8 条**，覆盖笔记全部核心知识点
- 每条以 `> - **` 开头，描述控制在 **80 字以内**
- 每条描述均为**独立条目**，禁止合并为长段落
- 列表项前有 `>` 符号（在 `[!summary]` callout 容器内）
- 禁止行为：将多条概念合并为一段文字描述

## 1️⃣ 完整知识库
[多个主题块，每个主题按四段式组织]

## 2️⃣ 修正与删除记录
[bug callout：记录所有修正和删除]

## 3️⃣ 代码库
[超过20行的综合代码案例，或注明"省略"]

## 4️⃣ 避坑指南 & 易错对比
[warning callout：易混概念对比表]
[danger callout：常见错误与规避]

## 5️⃣ 知识网络
[课内联动、前后衔接、AI/实战落地]

## 6️⃣ 应用场景与扩展（可选）
[与大主题相关但不属核心语法的内容，如行业案例、框架集成等。可自由控制是否包含代码示例]

## 7️⃣ 复盘与遗留（若原笔记有则保留并增强）
[掌握度自评、待巩固点、遗留疑问]

## 8️⃣ AI 附加说明
[组织方式、结构调整说明、重要性判断摘要、标注集中说明、难度标签分布、扩展块统计、代码库使用情况、可能遗漏主题、不确定项、自检声明]
```

**模块编号固定**：`1️⃣→2️⃣→3️⃣→4️⃣→5️⃣→8️⃣` 为强制模块，不得省略；`6️⃣→7️⃣` 为可选模块，无内容则删除（不保留空标题）。

## 二、主题块结构（四段式）

每个知识主题按以下四级标题组织：

```markdown
## 主题名 [难度标签]

### 定义与本质
[概念定义、核心机制、本质说明、包含关系等]

### 基础用法
[基本语法、核心参数、简单示例、常见用法]

### 进阶用法与原理（可选）
[深层原理、复杂用法、性能分析、算法推导]

### 避坑与局限
[常见误区、使用限制、注意事项]
```

### 2.1 难度标签规则

| 标签 | 含义 | 使用场景 |
|------|------|---------|
| 🔹 基础 | 术语定义、常识性概念 | 简单术语、基本概念 |
| 🔸 核心 | 核心算法、关键流程、必会知识 | 核心知识点、重点算法 |
| 🔺 难点 | 复杂推导、易混淆概念、深度理解 | 难点、易错点 |

### 2.2 AI 扩展块插入规则

扩展块使用 `[!note] 💡 AI 扩展（级别）` callout：

| 级别 | 触发条件 | 内容类型 |
|------|---------|---------|
| 基础 | 概念需要补充背景知识 | 背景解释、实际意义、应用场景 |
| 进阶 | 概念需要深入原理 | 算法原理、性能分析、工程实践 |

**插入位置**：紧跟在需要扩展的概念后面，通常在"定义与本质"或"基础用法"之后。

**数量规则**：
- 基础扩展：N≤3→1个, N=4~6→1~2个, N≥7→2~3个
- 进阶扩展：N≥3→1个, N≥6→1~2个
（N = 本笔记中独立知识主题的数量）

## 三、Callout 使用规范

| 类型 | 语法 | 使用场景 |
|------|------|---------|
| `[!summary]` | `> [!summary] 核心速览` | 开篇总览 |
| `[!note]` | `> [!note] 💡 AI 扩展（基础/进阶）` | AI 补充知识 |
| `[!tip]` | `> [!tip] 算法选择原则` | 提示/建议 |
| `[!info]` | `> [!info] 🧠 速查卡片` | API 速查 |
| `[!warning]` | `> [!warning] 核心理念` | 警告/重要理念 |
| `[!danger]` | `> [!danger] 常见错误与规避` | 严重错误规避 |
| `[!bug]` | `> [!bug] 修正记录` | 原笔记错误修正 |

## 附录A：主题内分层归纳指南

当单个主题下知识点较多时，使用中间分类节点承上启下：

```markdown
## 算法名 🔸 核心

### 定义与本质
### 基础用法
#### 分类1（如：参数说明）
#### 分类2（如：返回值说明）
### 进阶用法与原理
#### 算法原理
#### 复杂度分析
### 避坑与局限
```

**中间分类命名建议**：
- 参数说明 / API 说明 / 返回值说明
- 算法原理 / 数学推导 / 复杂度分析
- 应用场景 / 最佳实践 / 性能优化

### 何时可以省略某些层级

- **纯概念知识点**（如"数据分析方法论"）：可省略 L2，保留 L1 + L3（应用场景）+ L4
- **极简 API**（如 `len()`）：L1（定义）+ L2（示例）+ L4（注意点）即可，无需 L3
- **用户明确要求不套用四层结构时**：尊重用户偏好，不强制套用四层，但需在 `8️⃣ AI 附加说明` 列出原结构中被跳过的层级
- **用户笔记已有明显分层**：若用户笔记已自带明确层级（如使用了二级、三级标题且逻辑完备），请先尝试将现有层级映射到推荐的四层结构。若映射困难或会破坏连贯性，则直接保留并优化用户原结构，并通过 `👣 结构调整说明` 标注你做了哪些优化

### 统一格式规范

- 子标题使用 `###`（三级标题）
- 难度标签 `🔹 基础` / `🔸 核心` / `🔺 难点` 可放在子标题旁（如 `### 基础用法 🔹`）或段落开头（如 `🔹 基础：惰性计算`），但同一主题内需保持一致
- 若小节内只有一个代码块且无子标题，可直接跟在前文后

## 附录C：系列笔记深度均衡性

当用户要求增强同一课程/主题下的多篇笔记时，AI 应确保：

- 同类型算法（如均为分类算法）的扩展块数量差异不超过 1 个
- 核心概念（如损失函数、评估指标、sklearn API）的代码示例覆盖率应基本一致
- 若某篇笔记原内容特别简略，AI 应主动补充更多扩展块和代码示例，使其达到系列平均水平
- 系列笔记的难度标签分布应保持合理比例，避免全部为基础或全部为难点的极端情况

## 附录B：难度标签判定矩阵

| 判定维度 | 🔹 基础 | 🔸 核心 | 🔺 难点 |
|---------|--------|--------|--------|
| **理解深度** | 知道是什么 | 知道怎么用 | 知道为什么 |
| **记忆要求** | 名词+定义 | 流程+参数 | 原理+推导 |
| **出错代价** | 低（影响小） | 中（影响结果） | 高（导致错误结论） |
| **考试频率** | 偶尔考查 | 必考重点 | 拉分题目 |
| **对应处理** | 表格速查 | 代码示例+对比表 | 分步演算+扩展块 |

**判定流程**：
1. 该概念是否涉及复杂数学推导？→ 是 → 🔺 难点
2. 该概念是否是课程核心考点？→ 是 → 🔸 核心
3. 其他 → 🔹 基础
