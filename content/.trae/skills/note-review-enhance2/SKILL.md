# 学习笔记审核与增强

学习笔记审核与增强专家。对原始笔记进行完整留存、审核纠错、补齐增强与结构化输出，生成符合 Obsidian 语法规范的 Markdown 笔记。支持可选衍生输出：知识地图（Markdown）与思维导图（XMind）。

## When to Use

加载此 skill 的触发条件：

- 用户提供了一段学习笔记、课堂笔记或课程资料
- 用户要求"增强"、"审核"、"优化"、"整理"笔记
- 用户要求转换为 Obsidian Markdown 格式
- 用户要求生成知识地图、思维导图等衍生输出
- 用户反馈已生成的增强笔记有遗漏或错误，需要迭代更新

## When NOT to Use

- 用户仅要求翻译笔记（无增强需求）
- 用户要求从零创作新内容（非基于已有笔记）
- 用户提供的是已经高度结构化的文档（无需重组）
- 用户要求写代码/调试程序 → 使用 coding/debugging skills
- 用户要求分析数据 → 使用 data-analysis skills
- 用户要求生成图片/视频 → 使用 creative skills

## Quick Reference

| 输入 | 输出 | 触发条件 |
|------|------|---------|
| 原始笔记 `.md` | `N.主题（AI增强版）.md` | **生成模式**：用户要求增强笔记 |
| 原始笔记 + 增强版笔记 | 审核报告 + 修改建议 | **审核模式**：用户要求审核/比对/查漏补缺 |
| 已增强笔记 + 反馈 | 增量更新补丁 | **修复模式**：用户要求补充/修正/返工 |
| 多篇原始笔记 | `知识地图.md` | 用户要求知识地图 |
| 增强版笔记 | `[主题]思维导图.xmind` | 用户要求思维导图 |

**模式说明**：
- **生成模式**：从零生成增强版笔记，执行全部 6 个 Step
- **审核模式**：比对原始笔记与增强版，输出审核报告（优先执行 Step 2 概念完整性核对 + Step 5 语法/逻辑验收）
- **修复模式**：基于审核报告或用户反馈，做增量修改，保持原有内容不变

| 验收项 | 通过标准 |
|--------|---------|
| 语法验收 | 7项全部通过（参见 `references/obsidian-format.md`） |
| 逻辑验收 | 11项全部通过（含概念完整性） |
| 概念完整性 | 增强版 ≥ 原始笔记（不得简略） |
| 图片资源 | 迁移成功 + 验证存在 + 清理未引用 + 无占位符 |
| 代码块语言标注 | 所有 ``` 后必须有语言类型（python/text/mermaid 等），文本流程标注为 `text` |
| 版本号一致性 | 头部 `AI 审核版本` 必须与当前 Skill 版本一致 |

## Quality Gates

| 阶段 | 检查项 | 检查时机 | 失败处理 |
|------|--------|---------|---------|
| Step 1 | 图片迁移+验证+清理+编码规范化 | 输入预处理 | 悬空引用记录到修正记录；冗余图片删除；中文文件名建议重命名 |
| Step 2 | 概念完整性核对 | 内容增强前 | 回溯原始笔记，补全缺失概念 |
| Step 3 | 知识正确性+代码可用性 | 审核纠错 | 用 `[!bug]` 标注并修正 |
| Step 4 | 扩展块数量均衡性 | 补齐增强 | 补充至同级水平 |
| Step 5 | 语法验收（7项）+ 逻辑验收（11项）+ 代码块语言标注 + 版本号一致性 | 结构化输出 | 修复后重新自检 |
| Step 6 | 知识地图索引完整性 + XMind 编码/字体/class | 衍生输出 | 同步补充索引；修复编码和字体 |

## Core Pattern

### Step 1: 输入预处理
1. 提取原始笔记中的所有概念、定义、流程、代码、图片引用
2. **元数据推断**：若原笔记未标注日期/学习时长，尝试从文件修改时间或文件名推断；若无法推断，保留"未标注"并在"8️⃣ AI 附加说明"中提示用户补充
3. **重要性判断与清洗**：
   - **高重要性**：核心概念、关键代码、必要定义 → 必须保留
   - **中重要性**：扩展说明、重复但有助于理解的例子 → 可合并或精简
   - **低重要性**：明显无意义的占位符、与主题完全无关的内容、重复性语气词 → 删除
   - **所有删除或合并操作必须在"2️⃣ 修正与删除记录"中逐条说明理由**（空模板除外，见第6条）
4. 复制图片到 `asset/` 目录（迁移→验证→清理三步法）
   - **迁移**：将所有被引用的图片复制到 `asset/`
   - **验证**：扫描全部 `.md` 文件中的 `![[...]]` 引用，确认对应文件在 `asset/` 中存在；记录悬空引用
   - **清理**：扫描 `asset/` 目录，删除未被任何 `.md` 文件引用的图片；记录删除清单到 AI 附加说明
   - **引用路径规范化决策树**：
     ```
     图片引用格式判断：
     ├── ![desc](E:\绝对路径\xxx.png) → 复制到asset/ + 改为相对路径 ![desc](asset/xxx.png)
     ├── ![[xxx.png]] → 复制到asset/ + 保留WikiLink格式 ![[xxx.png]]
     ├── ![desc](asset/xxx.png) → 已正确，仅验证文件存在
     └── 原始有图但增强版无引用 → 评估是否迁移（关键流程图/时序图建议迁移）
     ```
5. **编码规范化**：若图片文件名含中文或特殊字符，必须重命名为英文+数字格式（如 `image_001.png`），避免跨平台编码问题；同步更新笔记中所有引用路径
6. 识别并删除空模板（无需记录）

### Step 2: 概念完整性核对（关键！）

**结构化核验步骤**（笔记数量 > 1 篇时强制执行）：

1. **提取原始笔记概念清单**：
   - 扫描原始笔记的 `###` 标题和 `**粗体术语**` 列表
   - 记录每个概念的定义/用法/关联/局限（四要素检查）
2. **提取增强版概念清单**：
   - 扫描增强版"1️⃣ 完整知识库"中的 `##` 主题标题和 `###` 子标题
   - 记录每个概念的覆盖情况
3. **差集比对**：
   - 列出"原始有但增强版无"的概念 → **必须补充**
   - 列出"原始有但增强版简略"的概念 → **回溯原始笔记扩写**
4. **逐条核验**：
   - 确认每个概念在增强版中有**等价或更详细**的表达
   - 确认"定义+用法+关联+局限"完整链条未丢失
   - 确认 A=B 对应关系（如"L1 正则化"对应"Lasso 回归"）已保留

**禁止行为**：
- 将正文描述压缩为 callout
- 仅保留名词不保留解释
- 删除流程步骤
- 丢失 A=B 对应关系

**若概念位置变动**，在"8️⃣ AI 附加说明"的"👣 结构调整说明"中标注

### Step 3: 审核纠错
- 知识正确性 → 用 `[!bug]` callout 标注修正
- 代码可用性 → 按三级标准处理：
  - **可独立运行**（≤10行、无需外部依赖）：提供运行结果注释 `# 输出：...`
  - **依赖内置数据**（如 sklearn 内置数据集）：正常保留，提供预期输出
  - **依赖外部文件**（CSV/图片等且无法运行时）：在代码块上方标注 `> [!warning] 本代码依赖外部文件：xxx.csv`
- 逻辑一致性 → 矛盾处用 `[!warning]` 标注
- **体系完整性**：检查知识点的前置依赖是否已在笔记中覆盖。若某概念依赖的前置知识未出现，在"5️⃣ 知识网络"中标注为"前置知识缺失：[[前置概念]]"，必要时在"1️⃣ 完整知识库"中补充极简前置说明
- 标注 ≥5 处时，正文内省略行内标注，集中到"AI 附加说明"

### Step 4: 补齐增强
- 缺失定义 → 补充
- **概念 → 代码直连**：
  - **需要代码示例的知识点**：语法规则、API 调用、函数用法、算法实现、复杂逻辑（递归、异步等）、易错模式、命令行
  - **无需代码示例的知识点**：纯概念定义、历史背景、应用场景描述、行业知识、方法论
  - 提供 ≤10行最小可运行示例，行内注释标注输出 `# 输出：...`
  - 必要时附典型错误对比（1种错误写法 + 错误输出 + 修正后代码）
- **重要函数/方法速查**：对笔记中涉及的重要函数/方法，嵌入速查卡片：
  - 签名、参数、返回值、异常
  - 输入→输出示例表（至少3组）
  - 参数效果对比（参数≥2时）
  - 场景标签：`🎯 最佳场景：`
- **复杂逻辑分步演算**：触发条件为递归、嵌套循环、继承链、异步、生成器链、代码>10行且复杂时，提供调用栈展开或状态变化表
- 易混概念 → 添加对比表（至少1组）
- 扩展块 → 按 N 值区间规则插入
  - 基础：N≤3→1个, N=4~6→1~2个, N≥7→2~3个
  - 进阶：N<3→至少选择1个知识点添加进阶扩展（兜底规则）, N≥3→1个, N≥6→1~2个
  - 每个扩展块必须包含 **原理/实战/性能/进阶** 中至少两项，可选补充：前沿发展、最新论文/工具动态、跨学科关联
  - 扩展块内**不包含完整函数或类实现**（可有一行代码片段，但不得是可运行的完整定义）
- **主动补充避坑指南**：若原笔记未提及常见错误，根据知识模式主动补充 1~3 条高频避坑规则，放入"4️⃣ 避坑指南 & 易错对比"模块
- **表格增强**：若原笔记包含对比表/参数表，检查表格完整性（表头是否清晰、行列是否对齐）；对关键表格补充行内注释或脚注说明；若表格内容存在矛盾，在"2️⃣ 修正与删除记录"中标注
- **数学公式增强**：对复杂公式（包含 ≥3 个变量或 ≥2 层嵌套运算），补充变量说明表（符号→含义→维度）；对推导型公式，在"进阶用法与原理"中补充关键变换步骤的文字解释
- **系列均衡性**：若某篇笔记扩展块数量明显低于同主题其他笔记，应主动补充至同级水平，确保系列笔记深度均衡
  - **优先级规则**：当系列均衡性与单篇 N 值规则冲突时，**系列均衡性优先**。允许单篇笔记的扩展块数量超出 N 值区间上限，但超出部分需在"8️⃣ AI 附加说明"中标注"因系列均衡性额外补充 x 个扩展块"

### Step 5: 结构化输出
- 严格按输出格式骨架组织（参见 `references/output-format.md`）
- 执行语法验收（7项）+ 逻辑验收（11项）
- **代码块语言标注检查**：扫描全文，确保所有 ``` 后均有语言标注；无标注的一律补标为 `text`
- **版本号一致性检查**：确认头部 `AI 审核版本` 与当前 Skill 版本一致
- 输出最终笔记 + AI 附加说明

### Step 6: 可选衍生输出
- 知识地图 → 参见 `references/mindmap-format.md` 第一章
  - **索引同步检查**：生成/更新知识地图时，必须扫描全部增强版笔记的核心概念，确保概念索引区覆盖所有笔记
  - **新增笔记处理**：若新增增强版笔记，必须补充对应的概念索引条目（至少包含：核心术语、关键流程、核心矛盾）
- 思维导图 → 参见 `references/mindmap-format.md` 第二章

## 规模化处理策略

当笔记数量 > 10 篇时，建议拆分为并行子任务：

| 笔记数量 | 分组策略 | 每组处理量 | 执行方式 |
|---------|---------|-----------|---------|
| 10-15 篇 | 按序号均分 3 组 | 4-5 篇/组 | 3 个并行 agent |
| 15-30 篇 | 按序号均分 5 组 | 5-6 篇/组 | 5 个并行 agent |
| >30 篇 | 先按主题分类，再每组 5-8 篇 | 5-8 篇/组 | 多轮并行 |

**分组原则**：
- 按学习顺序（day01-05、day06-10、day11-15）分组，保持知识连贯性
- 每组独立完成后汇总审核报告
- 跨组概念关联在汇总阶段统一检查

## Special Cases

| 场景 | 处理 |
|------|------|
| 原笔记明显错误 | 修正 + `[!bug]` 标注 + 记录到修正记录 |
| 原笔记缺失关键定义 | 主动补充 + 在 AI 附加说明中列出 |
| 原笔记代码无法运行 | 修正 + `[!bug]` 标注 + 提供运行结果 |
| 原笔记逻辑矛盾 | 按正确知识统一 + 修正记录说明 |
| 原笔记过于简略 | 要求用户补充信息，先输出已整理部分 |
| 原笔记缺少深度内容 | 确保扩展块总数达到 N 值区间规则的上限（基础+进阶），且至少包含1个进阶扩展 |
| 原笔记包含图片引用 | 迁移到 asset/ + 验证 + 清理未引用 |
| 原笔记大量空模板 | 自动过滤，无需逐条记录 |
| 无法确定知识点是否应包含 | 采用"宁可多补，不可遗漏"原则，在"AI 附加说明"中标注不确定性 |
| 原笔记以代码为主（>70% 为代码块） | 重点增强：逐行注释、复杂度分析、边界条件说明、多解法对比 |
| 原笔记以纯概念为主（无代码/无公式） | 重点增强：可视化类比、与实例的关联、应用场景举例、概念演化脉络 |

## Iteration

当用户对增强笔记提出补充/修正反馈时：

1. **分类反馈类型**：
   - 内容类（概念遗漏/错误）→ 回溯 Markdown 增强版修改
   - 结构类（层级/组织问题）→ 回溯 Markdown 增强版修改
   - 样式类（颜色/字体/布局）→ 针对 XMind/theme 调整
   - 格式类（文件打不开）→ 检查 XMind ZIP 结构
2. **内容/结构类**：增量生成扩展块，标注插入位置，保持原有内容不变
3. **样式/格式类**：记录变更到 Technical Notes，更新 XMind 生成脚本
4. **提供全量合并版本**（若用户要求）

## Common Mistakes

### AI 执行时常见错误

| 错误 | 后果 | 修复 |
|------|------|------|
| 将原始笔记正文描述压缩为 callout | 概念丢失，用户反馈"简略" | 正文必须保留完整描述，callout 仅用于扩展补充 |
| 仅保留名词不保留解释 | 知识点悬空 | 每个名词必须有定义或用法说明 |
| 删除原始笔记中的流程步骤 | 流程断裂 | 步骤必须完整保留，可重组但不可删除 |
| 丢失 A=B 对应关系 | 概念无法落地 | 如"L1正则化"必须对应"Lasso回归" |
| 图片只迁移不验证 | 引用失败 | 迁移后必须验证文件存在 |
| 未清理未引用图片 | 垃圾文件堆积 | 生成后扫描并删除未引用图片 |
| 保留 `[[filename.png]]` 占位符 | 显示失败 | 占位符必须删除或在不确定项中标注 |
| XMind manifest 放错位置 | 文件无法打开 | manifest.json 必须在 ZIP 根目录 |
| 给子节点加 `"class": "topic"` | 样式不生效 | 只有 rootTopic 加 class |
| 用 notes 字段存描述 | 描述不显示 | 描述直接作为子节点展示 |
| 核心速览写成长段落 | 无法快速扫描定位，与系列风格不一致 | 必须使用列表格式：`> - **概念名**：描述（≤80字）`，5-8条 |
| 代码块未标注语言类型 | 语法验收失败，Obsidian 渲染无高亮 | 所有 ``` 后必须跟语言类型，文本流程用 `text` |
| 代码块结束标记带语言后缀 | 语法验收失败，Obsidian 渲染异常 | 结束标记必须为纯 ```，禁止 ` ```python ` / ` ```bash ` / ` ```sql ` 等 |
| 版本号与 Skill 版本不一致 | 输出规范错误，后续迭代混乱 | **版本号铁律**：无论增量修改还是全文重构，版本号必须与 Skill 版本完全一致，不得递增 |
| 新增笔记后未同步知识地图索引 | 概念索引缺失，知识网络断裂 | 每次新增笔记必须扫描并补充知识地图索引 |
| XMind content.json 编码为 UTF-8 无 BOM | Windows 环境解析乱码 | 必须保存为 UTF-8 with BOM |
| XMind Theme 缺少中文字体 | 中文显示为系统默认字体（宋体） | Theme 中配置中文字体回退栈 |
| 系列笔记扩展块数量不均衡 | 部分笔记过浅，用户体验不一致 | 主动补充至同级水平 |

### 用户反馈常见模式

| 反馈 | 原因 | 处理方式 |
|------|------|---------|
| "概念简略了" | 原始笔记描述被压缩或删除 | 回溯到原始笔记，补回缺失内容 |
| "层级不够" | 中间分类节点缺失 | 增加"概述/特点/API"等分类节点 |
| "图片显示失败" | 图片未迁移或占位符未处理 | 检查 asset/ 目录 + 清理占位符 |
| "XMind 打不开" | 格式不兼容 | 检查 manifest 位置、metadata 版本、JSON 结构 |

## Technical Notes

- **图片复制**：Windows 某些目录 shutil.copy2 可能权限失败，使用 os.path.join + shutil.copy2 组合
- **XMind 2026 格式**：ZIP 必须包含 content.json + manifest.json（根目录JSON）+ metadata.json（dataStructureVersion:"3"），不得有 META-INF/
- **Obsidian WikiLink 误报**：`[[笔记名]]` 是内部链接不是图片引用，正则匹配时需过滤
- **XMind 节点规则**：rootTopic 加 `"class":"topic"` 和 `"structureClass"`，子节点不加
- **XMind 旧文件残留**：生成前必须 os.remove() 删除旧文件，否则 ZIP 中会残留旧格式文件导致报错
- **XMind 编码**：content.json 必须保存为 UTF-8 with BOM，避免 Windows PowerShell 解析乱码
- **XMind 字体**：Theme 中必须包含中文字体回退（Microsoft YaHei, PingFang SC, Noto Sans CJK SC），否则中文显示为系统默认字体
- **XMind class 验证**：生成后通过正则扫描 `content.json`，确认除 rootTopic 外，任何子节点均无 `"class":"topic"`
