## 1️⃣ 完整知识库

### NLP 概述 🔹 基础

#### 定义与本质

**NLP（Natural Language Processing，自然语言处理）**：让计算机去理解人类的语言。它是人工智能与语言学的交叉领域，目标是用计算机处理、理解并运用人类语言（如中文、英文），完成理解与生成两类任务。

**NLP 发展简史**：

```text
1950: 图灵提出"机器能够思考吗？"——划时代的话题
1957-1970: NLP 领域形成两大阵营：规则派 + 统计派
1994-1999: 基于统计的方法占据上风
2000-2008: 机器学习占据领导地位
2015-2023: 深度学习技术改革 NLP
2023 年至今: 大模型 AIGC 时代
```

发展脉络的核心规律是：**从依赖人工规则，逐步转向依赖数据驱动的统计/学习方法，再到端到端的深度学习与大模型**。

> [!note] 💡 AI 扩展（基础）
> **规则派 vs 统计派的本质差异**：
> - 规则派：依赖语言学家手写语法规则，可解释性强但覆盖面窄、维护成本高。
> - 统计派：从大规模语料中自动学习规律（如 N-gram、HMM），数据越多效果越好。
> 这一争论以统计派胜出告终，也奠定了今天"数据驱动"的 NLP 范式。理解这段历史，有助于明白为什么现代 NLP 高度依赖语料质量与规模。

#### 基础用法

**NLP 应用场景**：

```text
语音识别
机器翻译
搜索引擎
智能对话
……
```

这些场景可归为两大类：
- **自然语言理解（NLU）**：让机器"听懂/看懂"，如文本分类、情感分析、实体识别。
- **自然语言生成（NLG）**：让机器"说话/写文"，如机器翻译、对话回复、摘要生成。

#### 避坑与局限

- NLP 的难度高度依赖语言本身：==中文没有天然空格切分==，因此中文必须先分词，而英文可借助空格天然切分。
- 应用场景的效果受语料领域影响大，跨领域迁移往往需要重新训练或微调。

### 文本处理的基本方法 🔸 核心

#### 定义与本质

**文本预处理及其作用**：
- 借助 pandas 等第三方库对中文文本进行数据预处理，使其符合模型输入的要求。
- 文本预处理是 NLP 流程的前置环节，直接决定后续模型能拿到的特征质量。

**文本预处理的主要环节**：

```text
1. 文本处理的基本方法：分词、词性标注、命名实体识别
2. 文本张量的表示方法：one-hot、word2vec、wordEmbedding
……
```

这两个环节是层层递进的关系：先用"基本方法"把文本结构化（切成词、标注属性），再用"张量表示"把结构化文本数值化，最终才能喂给模型训练。

#### 基础用法

**（1）分词**

- **定义**：将连续的字序列，按照一定的方式重新组合成词序列的过程。
- **原因**：
  1. 模型训练之前必须先分词，单词是我们理解语义的基本单元；
  2. 英文具有天然的空格切分，但是中文没有。

**（2）命名实体识别（NER）**

- **定义**：
  - 命名实体：通常指人名、地名、机构名等专有名词。
  - NER：从一段文本中识别出上述描述的命名实体。
- **实现方式**：通过模型训练实现（后续项目实践中展开）。

**（3）词性标注**

- **定义**：对每个词语进行词性的标注，如动词、名词、形容词等。
- **实现方式**：使用 `jieba.posseg` 进行词性标注。

```python
import jieba.posseg as pseg
content = "我喜欢学习"
pseg.lcut(content)
# 输出：[pair('我', 'r'), pair('喜欢', 'v'), pair('学习', 'v')]
```

#### 避坑与局限

- 分词、NER、词性标注三者并非独立：分词是基础，NER 和词性标注都建立在分词结果之上。
- 中文分词存在歧义切分问题（如"结婚的和尚未结婚的"），单一工具难以完美解决，常需结合自定义词典或模型。

### jieba 分词工具 🔸 核心

#### 定义与本质

`jieba` 是 Python 中最常用的中文分词工具，支持多种分词模式与自定义词典。它通过词典 + 统计（HMM）的方式实现分词，兼顾准确率与速度。

#### 基础用法

**（1）精确模式**：按照人类擅长的表达词汇的习惯来分词，试图将句子最精确地切开，适合文本分析，也是默认模式。

```python
import jieba
content = "传智教育是一家上市公司，旗下有黑马程序员品牌。我是在黑马这里学习人工智能"
# 精确模式：cut_all 默认为 False
jieba.cut(content, cut_all=False)
# 返回一个生成器对象：<generator object Tokenizer.cut at 0x7f8d9053e650>

# 若需直接返回列表内容，使用 jieba.lcut 即可
jieba.lcut(content, cut_all=False)
# 输出：['传智', '教育', '是', '一家', '上市公司', '，', '旗下', '有', '黑马', '程序员', '品牌', '。', '我', '是', '在', '黑马', '这里', '学习', '人工智能']
```

**（2）全模式**：将尽可能成词的词汇分割出来，速度最快但存在歧义和冗余。

```python
jieba.lcut(content, cut_all=True)
# 输出：['传', '智', '教育', '是', '一家', '上市', '上市公司', '公司', '', '', '旗下', '下有', '黑马', '程序', '程序员', '品牌', '', '', '我', '是', '在', '黑马', '这里', '学习', '人工', '人工智能', '智能']
# 注意1：人工智能被全模式分成多个词（人工、智能、人工智能）
# 注意2：逗号和句号也被分成了词，且会出现空字符串 ''
```

**（3）搜索引擎模式**：在精确模式分词的基础上，将长粒度的词再次切分，提高召回率，适合搜索引擎分词。

```python
jieba.cut_for_search(content)
# 返回一个生成器对象：<generator object Tokenizer.cut_for_search at 0x7f8d90e5a550>

jieba.lcut_for_search(content)
# 输出：['传智', '教育', '是', '一家', '上市', '公司', '上市公司', '，', '旗下', '有', '黑马', '程序', '程序员', '品牌', '。', '我', '是', '在', '黑马', '这里', '学习', '人工', '智能', '人工智能']
# 对"上市公司""程序员""人工智能"等较长词汇都进行了再次切分
```

搜索引擎模式为何重要？下图展示了一个搜索召回的经典问题：当文档索引分词与查询分词不一致时，会导致"该召回的召不回"。

![[day01_search_recall.png]]

如图所示，doc1"黑马程序员"被索引为【黑马，程序，员】，若用户查询分词为【黑马，程序员】则无法完全匹配而召回失败；改用搜索引擎模式将查询也切分为【黑马，程序，员】即可匹配成功。这说明==索引侧与查询侧必须使用统一的分词策略==。

**（4）支持中文繁体分词**：

```python
content = "煩惱即是菩提，我暫且不提"
jieba.lcut(content)
# 输出：['煩惱', '即', '是', '菩提', '，', '我', '暫且', '不', '提']
```

**（5）支持用户自定义词典**：

词典的意义：如果用户指定了词典，那么 jieba 优先根据词典里面的词进行分词，避免专有名词被错误拆分。

词典的构造格式——每一行有 3 个元素：`word 词频（可省略） 词性（可省略）`：

```text
# 格式：word1 freq1 word_type1
黑马程序员 5 n
传智教育 6 n
人工智能 7 nz
学习 3
上市 3
```

代码实现：

```python
import jieba

sentence = '传智教育是一家上市公司，旗下有黑马程序员品牌。我是在黑马这里学习人工智能'
# 1 没有使用用户自定义词典
mydata = jieba.lcut(sentence, cut_all=False)
print('mydata-->', mydata)
# 输出：mydata--> ['传智', '教育', '是', '一家', '上市公司', '，', '旗下', '有', '黑马', '程序员', '品牌', '。', '我', '是', '在', '黑马', '这里', '学习', '人工智能']

# 2 使用用户自定义词典
jieba.load_userdict("./userdict.txt")
mydata2 = jieba.lcut(sentence, cut_all=False)
print('mydata2-->', mydata2)
# 输出：mydata2--> ['传智教育', '是', '一家', '上市公司', '，', '旗下', '有', '黑马程序员', '品牌', '。', '我', '是', '在', '黑马', '这里', '学习', '人工智能']
```

> [!warning] 本代码依赖外部文件：./userdict.txt（自定义词典，需按上述格式自行构造）

对比可见，使用自定义词典后，"传智教育""黑马程序员"被正确合并为整体，不再被错误拆分。

> [!info] 🧠 速查卡片
> **jieba 分词核心参数速查表**
>
> | 参数/方法 | 说明 | 典型值/示例 | 效果 |
> |-----------|------|-----------|------|
> | `cut_all=True` | 全模式开关 | `jieba.lcut(s, cut_all=True)` | 最细粒度切分，产生冗余词和空串 |
> | `cut_all=False` | 精确模式（默认） | `jieba.lcut(s)` | 符合人类习惯的最优切分 |
> | `HMM=True` | 启用 HMM 新词发现（默认） | `jieba.lcut(s, HMM=True)` | 能识别未登录词，但对已登录词无额外效果 |
> | `HMM=False` | 关闭 HMM | `jieba.lcut(s, HMM=False)` | 仅依赖词典，速度更快但不识别新词 |
> | `jieba.load_userdict(path)` | 加载自定义词典 | `jieba.load_userdict("./userdict.txt")` | 词典中的词优先切分，提升专有名词召回 |
> | `jieba.del_word(word)` | 动态删除词 | `jieba.del_word("黑马程序员")` | 将该词从词典中移除，后续可被拆分 |
> | `jieba.add_word(word, freq, tag)` | 动态添加词 | `jieba.add_word("黑马程序员", freq=10, tag='nz')` | 运行时临时加入词典，无需修改文件 |
> | `jieba.suggest_freq(seg, tune)` | 调节词频 | `jieba.suggest_freq(("黑", "马"), True)` | 强制"黑"和"马"分开，`tune=False` 强制合并 |
> | `jieba.cut_for_search(s)` | 搜索引擎模式 | `jieba.lcut_for_search(s)` | 精确模式基础上对长词二次切分 |
> | `jieba.tokenize(s)` | 返回词语位置 | `jieba.tokenize(s)` | 返回 (word, start, end) 元组，可用于高亮 |

#### 进阶用法与原理

- **三种模式的关系**：精确模式是基础；全模式在精确模式基础上把所有可能成词的都切出；搜索引擎模式则在精确模式基础上对长词做二次切分，本质是用更细粒度换取更高召回。
- **自定义词典原理**：jieba 内部基于词典 + 词频统计打分，加载自定义词典后会把这些词及其词频加入内部词表，使其在切分路径打分中胜出，从而优先成词。

#### 避坑与局限

- 全模式会产出空字符串 `''` 和过度切分，不适合直接用于文本分析，仅适合快速枚举候选词。
- `jieba.cut` 返回生成器，需用 `list()` 或 `jieba.lcut` 才能得到列表。
- 自定义词典的词频需合理设置，过高可能干扰其他正常切分。

### 文本张量表示概述 🔸 核心

#### 定义与本质

**文本张量表示的意义**：将文本转换为向量（数字）的形式，使得模型能够识别进而实现训练，一般是进行词向量的表示。

**实现的方式**：

```text
one-hot
word2vec
wordEmbedding
```

三者是层层演进的关系：One-Hot 是最朴素的离散表示；Word2Vec 是通过无监督训练得到的稠密表示；WordEmbedding 是更一般的"词嵌入"思想（可由 nn.Embedding 等实现）。

**One-Hot 词向量表示定义**：针对每一个词汇，都会用一个向量表示，向量的长度是 n，n 代表去重之后的词汇总量，而且向量中只有 0 和 1 两种数字。俗称：独热编码、01 编码。

#### 基础用法

以一个 5 词的小词表为例，One-Hot 编码如下：

```text
我      -> [1, 0, 0, 0, 0]
爱      -> [0, 1, 0, 0, 0]
黑马    -> [0, 0, 1, 0, 0]
程序员  -> [0, 0, 0, 1, 0]
啊      -> [0, 0, 0, 0, 1]
```

每个词对应一个长度为词表大小 n 的向量，只有该词所在位置为 1，其余为 0。

#### 进阶用法与原理

Word2Vec 与 WordEmbedding 不再用稀疏的 0/1 向量，而是把每个词映射到一个低维稠密空间，让语义相近的词在向量空间中距离相近。下图展示了 Word2Vec 的 CBOW 模型如何用上下文预测中间目标词，并将模型权重参数当作词向量的表示。

![[day01_nlp_01.png]]

如图，CBOW 的核心思想是"上下文预测中间目标词"：输入层接收上下文词的 One-Hot 向量，经权重矩阵 W 投影到隐藏层得到稠密词向量，再经输出层 W' 与 softmax 预测中心词。==训练得到的权重矩阵 W 的每一列就是一个词的词向量==。

而词嵌入的本质是"查表"操作：用 One-Hot 向量与权重矩阵相乘，等价于提取矩阵中对应列，如下图所示。

![[day01_nlp_02.png]]

如图，One-Hot 向量充当"选择器"，与嵌入矩阵相乘后提取出对应词的稠密列向量；多个词的向量还可通过求和（Sum Pooling）等方式聚合成句子表示。这正是后续 nn.Embedding 层的工作机制（详见 [[2.文本张量表示方法（AI增强版）]]）。

> [!note] 💡 AI 扩展（进阶）
> **为什么稠密词向量优于 One-Hot？**
> - **原理**：One-Hot 的任意两个不同词向量正交（内积为 0），无法表达词与词的相似度；Word2Vec 通过预测任务学习到的稠密向量，使语义相近的词在空间中靠近（如 `vec(国王) - vec(男) + vec(女) ≈ vec(女王)`）。
> - **性能**：One-Hot 维度等于词表大小 n（动辄数万到数十万），存在维度爆炸与稀疏问题；Word2Vec 用 50~300 维即可紧凑编码语义，存储与计算开销大幅降低。
> 因此实际工程中几乎不直接用 One-Hot 训练模型，而是用预训练或端到端学习的词嵌入。

#### 避坑与局限

- **One-Hot 编码的缺点**：
  - 割裂了词与词之间的联系（主要缺点，任意两词正交）；
  - 如果 n 过大，会导致占用大量的内存（维度爆炸）。
- One-Hot 仅适合教学演示或极小词表场景，真实任务应使用 Word2Vec / WordEmbedding。

## 4️⃣ 避坑指南 & 易错对比

> [!warning] 核心理念
> **分词模式选择原则**：文本分析用精确模式；快速枚举候选词用全模式；搜索引擎/召回场景用搜索引擎模式。==模式选错会直接影响后续模型效果与召回率==。

| 对比维度 | 精确模式 | 全模式 | 搜索引擎模式 |
|---------|---------|--------|------------|
| 调用方法 | `jieba.lcut(s, cut_all=False)` | `jieba.lcut(s, cut_all=True)` | `jieba.lcut_for_search(s)` |
| 切分粒度 | 最精确，符合人类习惯 | 最细，尽可能成词 | 精确 + 长词二次切分 |
| 是否产生空串 | 否 | 是（`''`） | 否 |
| 适用场景 | 文本分析（默认） | 候选词枚举 | 搜索引擎索引/查询 |
| 对"人工智能" | 保留整体 | 拆为"人工/智能/人工智能" | 拆为"人工/智能/人工智能" |

> [!danger] 常见错误与规避
> - **误用全模式做分析**：全模式会产生空字符串和过度切分，直接用于模型训练会引入噪声。规避：文本分析一律用精确模式。
> - **忘记 `cut` 返回生成器**：`jieba.cut()` 返回生成器对象，直接 `print` 看不到结果。规避：用 `jieba.lcut()` 或 `list(jieba.cut(...))`。
> - **自定义词典未生效**：词典文件路径错误或格式不规范（词频/词性列缺失未用空格分隔）。规避：确认 `load_userdict` 路径正确，每行用空格分隔。
> - **One-Hot 直接训练**：词表大时维度爆炸且词间无相似度。规避：真实任务改用 Word2Vec / nn.Embedding。
